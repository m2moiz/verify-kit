<objective>
Land the release-please release automation (DOC-02 / D-10): config file, manifest, GitHub Actions workflow, CHANGELOG.md stub. release-please will open release PRs from conventional-commits on every push to main, generating CHANGELOG sections (Added / Fixed / Changed / Documentation). The operator hand-edits the "Breaking changes for consumers" block in each release PR before squash-merging.

Purpose: Establishes SemVer enforcement via the commit-message contract (D-12) and a consumer-facing migration callout per release (D-11). This plan ships in W1 alongside 06-01 (no dependencies); 06-06 README and 06-07 CONTRIBUTING reference the CHANGELOG path and the commit-message contract this plan locks.

Output: Four new repo-root / workflow files.
</objective>

<constraints>
- D-10: release-please (Google) is the release automation.
- D-11: per-release "Breaking changes for consumers" callout is mandatory; hand-edited on the release PR before merge.
- D-12: SemVer enforced via release-please commit-message contract; documented in CONTRIBUTING.md (06-07 owns that doc).
- §1 Research: pin `googleapis/release-please-action@v4` (not `@main`); use manifest mode; `bump-minor-pre-major: true` + `bump-patch-for-minor-pre-major: true` for pre-1.0; permissions `contents: write` + `pull-requests: write`; no custom token (default GITHUB_TOKEN is sufficient for a template repo with no downstream-trigger needs); changelog-sections override emits Added/Fixed/Changed/Documentation.
- §1 Pitfall: `release-as:` override is forbidden in this plan (would force a version and bypass SemVer detection).
- §12 Research: "Breaking changes for consumers" is consumer-facing migration instruction, NOT just a commit-log dump. Empty section reads `_None — \`copier update\` is safe and silent._`.
- Open Question 7 (§Open Questions): start manifest at `0.0.0` so the first `feat:` commit triggers the 0.1.0 release PR. Verified per §1 manifest mode semantics.
- REVIEW-CHECKLIST §1: no relative paths in the workflow (act-validated separately in 06-08).
- REVIEW-CHECKLIST §5: workflow YAML is plain (no Jinja); no inline conditional risk.
- act-local validation: workflow can be listed via `act -l push`; full execution requires GITHUB_TOKEN secret which is not available locally — document this caveat in the SUMMARY.
</constraints>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → repo write access | `contents: write` + `pull-requests: write` permissions granted; scoped to release PR creation only |
| Conventional-commit messages → CHANGELOG generation | Commit-message parser is trusted; malicious commit messages could inject CHANGELOG content but cannot escalate privileges |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-14 | Tampering | Bad actor pushes a `feat!:` commit that bumps version unexpectedly | accept | Solo-dev project; commits to main require PR; PR review catches malicious commit-message shapes |
| T-06-15 | Elevation of Privilege | release-please action pinned to a moving tag (`@main`) gets compromised | mitigate | Pinned to `@v4` per §1 (matures more slowly; major-version pin is the canonical GitHub Actions hygiene) |
| T-06-16 | Repudiation | CHANGELOG entry written incorrectly without operator review | mitigate | release-please opens a PR (not a direct push) — operator must explicitly merge; hand-edit step for Breaking changes forces a review |
</threat_model>

  <action>
    Two-part fix:

    **(a) `template/justfile.jinja2` `verify-backend` recipe:** wrap the docker-up + schemathesis-fuzz + docker-down sequence so docker-down ALWAYS runs even on mid-recipe failure. The cleanest just-recipe shape:

    ```
    verify-backend:
        #!/usr/bin/env bash
        set -e
        trap 'docker compose -f docker-compose.yml -p scratch down -v --remove-orphans 2>/dev/null || true' EXIT
        docker compose -f docker-compose.yml -p scratch up -d
        # ... existing schemathesis/curl invocations ...
    ```

    The `trap` ensures `docker compose down` runs on every exit path (success, failure, Ctrl-C). The `-p scratch` project name allows the test to grep `docker ps --filter name=scratch` for orphan detection.

    **(b) `tests/test_phase04_integration.py`:** add a new test `test_verify_backend_full_path_leaves_no_orphan_containers`:
    1. Render a scratch project with `has_backend=true, has_db=true`.
    2. Run `just verify-backend` inside it (use `subprocess.run` with `env=_CLEAN_ENV`).
    3. Regardless of exit code, run `docker ps --filter name=scratch --format '{{.Names}}'`.
    4. Assert the output is empty (no orphan containers).
    5. Also test the failure-cleanup path: deliberately break a step (e.g., write a syntactically-invalid docker-compose.yml.bak override), run again, assert STILL no orphans.

    Mark the test `@pytest.mark.skipif(shutil.which('docker') is None, reason='docker not available')` so it skips cleanly in environments without Docker.
  </action>
  <verify>
    <automated>uv run pytest tests/test_phase04_integration.py::test_verify_backend_full_path_leaves_no_orphan_containers -v 2>&1 | tail -8</automated>
  </verify>
  <done>Test passes; `docker ps --filter name=scratch` empty after both success and failure paths of `just verify-backend`.</done>
